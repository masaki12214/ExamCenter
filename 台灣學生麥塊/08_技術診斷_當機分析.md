# 08 · 技術診斷：07-14 Watchdog 當機分析

- **資料來源**：使用者上傳崩潰 log（`bd8ce160-202607144.log`，涵蓋 17:32:42 – 19:28:47）
- **診斷時間**：2026-07-14

## 結論
**官方商店 + 自製經濟每次交易同步寫全表 MySQL，加上執行緒開到爆，主線程卡死被 Paper Watchdog 強制關服。**

## 事發經過
19:27:01 Paper Watchdog 偵測主線程凍結 >10 秒，建立 thread dump 並於 19:28 強制關閉。

### 1) 直接凍結主線程：自製經濟同步寫 MySQL
Watchdog 主線程堆疊：
```
me.hina.tsm.gui.OfficialShopCategoryGui.lambda$onOpen$2(OfficialShopCategoryGui.java:153)   ← Hina 商店 GUI 點擊
 → com.example.vault…SimpleEconomy.depositPlayer(SimpleEconomy.java:289)
   → SimpleEconomy.save(SimpleEconomy.java:425)
     → Database.saveAllBalances(Database.java:96)    ← 每次交易「全表餘額」同步寫入
       → MySQL socket read … 阻塞
```
玩家在官方商店點擊 → 經濟 `depositPlayer` → **全表餘額同步寫 MySQL、寫在主線程**。DB 一慢即凍結全服。

### 2) 底層病根：執行緒爆炸
- **13 × `OutOfMemoryError: unable to create native thread`**（皆發生在 `Thread.start0`）＝ 撞到容器 thread/process（nproc/pids）上限，**非** heap 不足。
- Craft Scheduler 執行緒**開到 469 號、同時存在 221 條**（正常應為個位數）＝ 明顯執行緒洩漏。
- 撞上限後 eco、InventoryRollbackPlus、InteractiveChatDiscordSrvAddon 等皆無法再開執行緒；連 MySQL 連線可能都開不出，導致上述 DB 查詢卡死。

其他異常：ConcurrentModificationException ×11、StackOverflowError ×8、NullPointerException ×5。

## 「vault-1.6.5-java21.jar」真面目
`plugin.yml`：`name: Vault`、`main: net.milkbowl.vault.Vault`、
`description: Internal Economy provider like Vault 2.0 (no Vault.jar dependency)`，
含指令 balance/pay/vaultop/vault/loan/eco。
→ **冒用 Vault 名稱與套件路徑，但自帶整套經濟**（真正存錢邏輯在 `com.example.vault.economy.SimpleEconomy` / `storage.Database`）。
**不是官方 MilkBowl Vault**（官方 Vault 只是橋接 API，不存錢、不碰 MySQL）。`com.example` 為佔位命名，屬自製／第三方。

## 解法
**核心（改經濟）：**
1. `depositPlayer` 不要每次 `saveAllBalances()` 全表；改成只存該玩家 + **async 非同步**。
2. DB 用固定大小連線池（HikariCP），勿每筆交易開新連線／執行緒（＝執行緒爆炸主嫌）。
3. Hina 官方商店 GUI 點擊也改 async。

**環境層面（native-thread OOM 正解）：**
- ⚠️ 不要用「加記憶體」解；`-Xmx` 反而別設太滿，留 off-heap 給執行緒堆疊。
- 請主機商 athebyne 調高容器 `nproc` / `pids` 上限（現況 swap=0）。

**替代方案**：改用有維護的 **VaultUnlocked** + 正規經濟（CMI 內建經濟 或 XConomy），並遷移餘額資料。
